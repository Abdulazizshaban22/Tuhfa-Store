# Runbook — HTTP 5xx > 5% أو P95 > 500ms

## 1) تحقّق بسرعة
- افتح Grafana لوحة `Admin / APIs` → Panels: HTTP errors, Latency.
- حدّد الفترة الزمنية (آخر 15m) وابحث عن تغييرات نشر/ذروة مرور.

## 2) تشخيص
- إذا كانت الأخطاء **5xx**:
  - افحص سجلات API: `kubectl logs -l app=tohfa-api --tail=200`
  - اختبر نقطة `/health` و `/metrics`
  - راجع تغييرات قاعدة البيانات (مؤشرات، خطط استعلام)
- إذا كان **P95 latency** مرتفع:
  - راقب قواعد ORM البطيئة، مؤشرّات مفقودة، أو ضغط خارجي (upstream).

## 3) استعادة سريعة
- فعّل التوسّع الأفقي (HPA) مؤقتًا،
- أعد تشغيل Pod مُتأثر،
- عطّل ميزة تجريبية عبر Feature Flag إن وجدت (`/admin/flags`).

## 4) الإغلاق
- اكتب سبب الجذر (RCA) + إجراء وقائي، واربط التنبيه في Slack بـ PR الإصلاح.
